Color Dash

by: Sohan Agarkar

This game was made for my level 2 project at The League of Amazing Programmers.
Click here to download and play!

1. Why I Made This Game

I decided to choose this game design over my other designs because it presented some challenges that I thought would be interesting to solve. It was also much more original than my other designs. The game also allowed me to learn more about many aspects of Object Oriented Programming that I wouldn't have otherwise used. For example, I had to use and learn about sub-classes as I was making the game.

2. What I Learned

From user testing, I learned that my game was much more difficult for others than it was for me, so I slightly decreased spawn rates to recognize the issue. The first tester also had some issues with the controls, which I fixed by adding a quick description of how to play the game.

3. Architecture of My Game

The ColorDash class creates a frame and the Game Panel class manages everything about the game. The EndlessObjectManager class, an object of which is made by the Game Panel, manages all of the elements of the actual gameplay like the player, obstacles, and streets. The Street class manages its lanes and vehicles that drive on it via a sub-class. All collision interactions are handled within the EndlessObjectManager object.

4. Technical Implementation

My game has several classes. The basics are the Runner and the Game Panel. The Runner holds the frame and instantiates a Game Panel. The Game Panel essentially manages everything on the main menu and instantiates a new EndlessObjectManager every time a new game is started. The Game Panel manages key presses, instructions, high scores, etc. The EndlessObjectManager managed all of the elements of the game. This included the player object (which was of the Soldier class), the Street objects, score counts, collisions, etc. Every Street class had a sub-class called LaneTraffic. This class manages one "lane" (or a row of endless vehicles going in one direction) of vehicles on a Street, as a Street can have multiple lanes. LaneTraffic moved the vehicles (which were of the Obstacle class), deleted them when they were off the screen, and much more. The actual Obstacle class itself simply managed its collision boxes and decided which graphic to use based on the direction, type, and color of the vehicle. I am particularly proud of the LaneTraffic sub-class as it taught me much about how such a class can access the rest of the game and, with the position of the sub-class in mind, which classes certain functionalities should go into to maintain a minimal amount of variables and loops in the program.

5. Most Challenging Part

The most challenging part of this project was the Obstacle lanes. The initial lane manager I created was simply a function in the Street class. This approach limited the flexibility of the amount of different ways a Street object could be created. Along with this, Obstacles seemed to constantly overlap and occasionally go in the wrong direction. It took me a while to think about using a sub-class within Street to manage Lanes. When I did, it only took an hour or 2 to get a new working Lane structure.

6. Future Fixes and Updates

If I were to continue working on this game, I would make a graphic for the Street, so it doesn't look like vehicles are coming from an angle. I would also make a new class to manage an interactive tutorial, so players can have a hands-on experience with the game's mechanics before moving onto the main product. I am sure that users would prefer a full-on tutorial rather than reading the instructions.
